# Decisions — Architecture & Library Choices (Choice / Alternative / Reason)

- **Working conventions home = `CLAUDE.md`** / Alt: AGENTS.md, memory-only / Reason: CLAUDE.md is auto-loaded by Claude Code each session and is version-controlled, so it reliably enforces agent behavior. Rejected AGENTS.md (it is the challenge's given canonical contract — mixing our process into it pollutes the contract). Rejected memory-only (not in repo, not portable across machines, can be dropped in long-conversation summarization). Memory kept only as redundant fallback.
- **Persistent docs live under `code/docs/`** / Alt: repo-root `docs/` / Reason: only `code/` is zipped into `code.zip`, so docs under `code/docs/` reach the AI Judge; repo-root docs would be excluded from submission.
- **Eval framework folder = `code/evaluation/`** / Alt: `code/eval/`, or `code/eval/` framework + `code/evaluation/` shell / Reason: `problem_statement.md` explicitly mandates an `evaluation/` folder and `evaluation/evaluation_report.md`, and the starter ships `code/evaluation/main.py`; when the spec names a path, follow it. Single folder is more efficient than maintaining a separate `eval/` framework plus an `evaluation/` shim. Supersedes the user's earlier preference for the name `eval` (per user instruction: follow explicit naming spec).
- **Shared schema = `code/schema.py`, single source of truth for allowed values** / Alt: duplicate the vocabularies in eval and in the system separately / Reason: both the eval validator and the production system must agree on the exact column order and allowed-value vocabularies; one shared module prevents drift, so adding/changing a field is a one-place edit.
- **Eval consumes a predictions CSV; it does NOT run the VLM** / Alt: eval directly drives the system end-to-end / Reason: decoupling keeps the scorer deterministic, zero-cost, and repeatable; it can score any predictions file (enabling the README-required ">=2 strategy" comparison by feeding multiple pred files against one truth), and system outputs can be cached/reused.
- **Free-text fields not auto-scored in v1 (`*_reason`, `claim_status_justification`)** / Alt: LLM-judge now / Reason: LLM judging adds cost, nondeterminism, and build time; v1 focuses the signal on the structured fields (headline = `claim_status` accuracy). A pluggable hook is left so an LLM-judge can be added later without reworking the framework.
- **Predicted<->expected join key = `image_paths`** / Alt: `user_id` / Reason: `user_id` repeats across rows in the test set (e.g. user_004/018/034/040/041/042/045 appear multiple times) so it is not unique; each `image_paths` maps to a unique case directory, making it the stable alignment key.
- **Modality = pure VLM first, add thin deterministic CV helpers on demand** / Alt: train/fine-tune specialized vision models; fully separate vision-tools + text-LLM (LLM never sees pixels) / Reason: the core judgment is grounding visual evidence against the claim's part/issue/severity, which requires joint vision+language reasoning (VLM's strength) and has no off-the-shelf model matching the 12-issue x per-object-part taxonomy. Rejected fine-tuning (only 20 labeled rows, 24h — infeasible). Rejected fully-separate pipeline (intermediate text labels are an information bottleneck for mismatch/severity cases). Start pure VLM (simplest), then add cheap deterministic CV (Laplacian blur->blurry_image, exposure->low_light_or_glare, OCR->text_instruction_present + untrusted-text feed, EXIF->non_original_image) only where eval shows a bottleneck and where determinism is wanted ("Deterministic where possible").
- **Control flow = fixed pipeline (static DAG), NOT ReAct / agentic tool-loop** / Alt: ReAct reason-act-observe loop / Reason: every claim's steps are known and fixed (extract claim -> visual check -> join history -> fuse), so no dynamic planning is needed. Pipeline gives reproducible, countable model calls (serves "Deterministic where possible" + the operational-analysis cost report), is parallelizable across the 44 independent claims, maps stages to fields for single-variable debugging, and minimizes the injection attack surface (the agent takes no consequential actions, only emits structured data). A bounded validate-and-repair retry is allowed (still pipeline, not ReAct).
- **Model behind a `VLMClient` abstraction; skeleton built against a `StubVLMClient`** / Alt: hardcode one vendor/model into the pipeline / Reason: model selection then does not block skeleton work, the whole pipeline is testable end-to-end with a deterministic stub (no API/cost), and swapping clients gives the README-required ">=2 model/strategy" comparison for free. Concrete model choice + current pricing deferred to when the real client is wired (pin via the API reference, not from memory).
- **Cache key includes `model_id` + `prompt_version`** (plus prompt-text hash + image set) / Alt: key on image set + prompt only / Reason: without model_id in the key, switching models would silently reuse another model's cached responses and contaminate experiments; per-(model,prompt) caches keep runs comparable and make model switches clean.
- **Every run emits run-metadata; experiments stay single-variable and resumable** / Alt: keep only the predictions CSV / Reason: tagging each run with model_id, prompt_version, call/cache counts, timing, timestamp, git commit makes results traceable and feeds the operational analysis. Combined with the single-variable rule (change only model OR prompt between two logged runs) and a resumable cache (rate-limit hits leave completed cells cached; resume completes the run), this prevents "half model A / half model B" dirty data when limits or model changes occur.
- **Integration via a native per-vendor SDK client, NOT an OpenAI-compatible gateway** / Alt: OpenAI-compatible gateway (OpenRouter etc.) / Reason: native SDKs expose three efficiency/accounting levers that gateways typically drop or degrade — prompt caching (cache reads ~0.1x input price on the large stable system+vocab prefix reused across all 44/85 calls), a Batch API (~50% off; the test set is not latency-sensitive), and precise per-call usage fields (cache read/write token breakdown). At this scale absolute cost is cents, so the deciding factor is the SCORED operational-analysis section (it asks us to demonstrate batching/caching/cost) plus accurate accounting for `evaluation_report.md`, which native gives cleanly; a gateway also risks markup. **Scope: this fixes only the integration STYLE (native vs gateway). The concrete vendor (Anthropic / OpenAI / other) and model are still OPEN.** Whichever vendor is chosen later, implement its single native client class behind the existing `VLMClient` interface.
- **Develop on the ship-tier model throughout; model choice is an eval-driven variable** / Alt: phased "weak model during dev, switch to the best only at the end" (cost-saving) / Reason: at this scale a full run is cents and the disk cache (keyed by model_id+prompt_version) makes repeated iteration ~free, so the cost premise behind phasing evaporates. A weak dev model confounds the signal — bottleneck misattribution (model-ceiling errors look like prompt bugs), over-fitting scaffolding the ship model won't need, and inability to tell prompt-ceiling from model-ceiling; and a blind late switch ships output.csv on a model the prompt was never tuned against (can regress). Instead: pick the model empirically via an early sweep (held-constant prompt), develop on that ship-tier model, and re-confirm/lock with a late sweep on the tuned prompt; if the final model differs, do a short re-tune rather than a blind switch.
- **README "≥2 configurations" = compare any of strategy / prompt / model (an OR, a floor); architecture decisions are judged single-variable on the FIXED model** / Alt: design a multi-model grid to drive architecture decisions / Reason: the requirement is satisfied automatically — every single-variable iteration in experiment_log.md is a 2-config comparison (floor vs MVP, prompt v1 vs v2, arch A vs B). Architecture upgrades must be judged holding the model fixed (clean attribution; and we develop on the ship model anyway, so that is the right place to judge them). Reserve multi-model comparison for two narrow purposes only: (1) model selection (early + late sweep), (2) an optional end-of-project generalization check (does the tuned system's advantage hold across models — good for the report/AI-Judge, not for steering architecture). Avoid a model x prompt x architecture grid (combinatorial, confounded); default to changing one axis at a time.
- **Core VLM = Gemini (native SDK); DeepSeek is not a VLM -> not usable for the core** / Alt: set up Anthropic/OpenAI now; use DeepSeek / Reason: the task is vision-first (images are the source of truth), so the core model must accept image input. Of the two keys the user has set up right now, only Gemini is a native multimodal VLM; DeepSeek's API is text-only (its DeepSeek-VL weights are not on the platform API), so it cannot do the visual judgment. Gemini is fully sufficient for this task, so no need to set up another vendor. DeepSeek is kept as an OPTIONAL text-only secondary role (e.g. candidate-B stage-1 claim extraction, or a text-side comparison point) — not introduced in the MVP. This resolves the previously-open vendor slot to Gemini *given current resources*; if another vendor is set up later it can be added to the model sweep.
- **Env isolation via a uv-managed `.venv` (python 3.12); canonical manifest = `requirements.txt` (pinned)** / Alt: system python; pyproject + uv.lock only; poetry/conda / Reason: rules require excluding virtualenvs from the zip (README) but mandate no tool, while requiring a runnable/reproducible solution — so the evaluator-facing artifact must be a universal `pip install -r requirements.txt`, not a uv-only lock (we don't control the evaluator's toolchain). uv is used locally only (fast, already installed); `.venv` is gitignored and never shipped. Python pinned to 3.12 (not the system's 3.14) for dependency wheel availability. Versions pinned after first successful install (`google-genai==2.9.0`) for reproducibility.
- **Secrets via a repo-root `.env` loaded by a guarded `python-dotenv` `load_dotenv()`** / Alt: shell `export` only; `uv run --env-file`; hardcode / Reason: AGENTS.md endorses a `.env` file, and a `.env` keeps real keys out of the chat transcript and out of git entirely (vs `export`, which only keeps them out of code). `load_dotenv()` in main.py is wrapped in try/except so it is a no-op when python-dotenv is absent or no `.env` exists (the evaluator can just set env vars directly). `.env` is gitignored; a committed `.env.example` documents the required `GEMINI_API_KEY` (falls back to `GOOGLE_API_KEY`).
- **Sequencing refinement: tune the prompt BEFORE the model sweep, not after** / Alt: the earlier "early sweep to pick the dev model first" plan / Reason: EXP-002 showed the two worst fields (severity 20%, issue_type 40%) are prompt under-specification, not model capability — the prompt gives severity zero level-definitions and issue_type no disambiguation of confusable types. A flash-vs-pro sweep on a zero-definition prompt would mostly measure "which model guesses the undefined rubric better" (noise). So: fix the prompt on the fixed dev model (gemini-2.5-flash) first, then run ONE model sweep later on the improved prompt (meaningful comparison + ship-model lock; the "≥2 configurations" requirement is already met by prompt v1 vs v2). Residual issue_type errors after definitions are added (e.g. crack vs glass_shatter has a perceptual component) are exactly what that later sweep will adjudicate.
- **Results recorded as per-run JSON (`code/evaluation/reports/*.json`) + an auto-generated scoreboard (`code/docs/results.md` via `evaluation/scoreboard.py`)** / Alt: hand-maintained results.md table / Reason: the comparison table is the factual support for model/prompt decisions, so it must not drift from the scorer — hand-transcribing eval numbers is a real error source. eval already emits the JSON (`--json --tag`), so this is not new machinery, just persisting an existing artifact + a small read-only aggregator (no API, no model runs — stays within the eval-consumes-CSV decoupling). Keeps the framework lean while removing transcription error; results.md is generated, not edited by hand. See [[lean-artifacts]].
- **Drop gemini-2.5-pro; dev model stays gemini-2.5-flash; gemini-3.5-flash is an open candidate** / Alt: adopt pro, or switch to 3.5-flash now / Reason: EXP-004 sweep (prompt v2) showed pro gives no benefit over flash (same headline 75, worse object_part) at higher cost — rejected. 3.5-flash wins on perception fields (issue_type 50, object_part 90, severity 60) but loses on the HEADLINE claim_status (65 vs 75); since claim_status is the core decision, flash stays the dev model. 3.5-flash remains an open ship candidate pending an investigation of why its claim_status is lower — if recoverable, it likely becomes the ship model. Model choice between flash and 3.5-flash is NOT yet final.
- **Ship/dev model committed: `gemini-3.5-flash` · prompt v3** / Alt: gemini-2.5-flash · v3 / Reason: EXP-005 error analysis showed the claim_status gap (flash 75 vs 3.5 70) is essentially ONE non-shared case — both models miss the SAME 3 hard adversarial cases (007 blurry, 014 damage-not-visible, 020 seal-text) that cap the headline — so decision quality is ~equivalent, while 3.5-flash has a clear perception lead (issue_type 55 vs 40, severity 70 vs 50, object_part 90 vs 85) and the best flat average (72.1 vs 69.6). pro already dropped. `main.py` defaults set to this config so the bare entry point (`python code/main.py`) produces output.csv with the ship config.
- **Fix 1 (same-model grounding-verification pass) REJECTED — net-negative** / Alt: keep `--verify` on / Reason: EXP-006 — a second call to the SAME model shares its perceptual blind spots (correlated errors): it failed to catch the target over-support (014) and falsely flipped a true `supported` case (001), netting claim_status 70->65. `--verify` left off by default (revisit only with an INDEPENDENT/different-family verifier, which we don't have). The over-support ceiling on subtle no-damage cases is accepted as a model-family limit; stop chasing 3 sample ceiling-cases with same-model tricks (overfit + cost risk).
- **L2-lite multi-image rule ADOPTED (prompt v4); ship config = gemini-3.5-flash · v4** / Alt: per-image multi-call decomposition (L2-full) / Reason: EXP-007 — a single-call "any-image-suffices" rule lifted claim_status 70->80 and supporting_image_ids 69->82 with ZERO extra calls; chose it over per-image decomposition (which would ~2x calls, conflicting with the cost constraint [[token-efficiency]]). This also corrects EXP-006's premature "model ceiling" framing — case_007 was a harness multi-image gap, not perception. Minor regressions (severity -10, issue_type -5) are outweighed by the headline + flat-avg gain (72.1->76.1); severity is a watch item.
- **Pin `temperature=0` on Gemini calls (reproducibility)** / Alt: leave Gemini's default (stochastic) sampling / Reason: a severity-ONLY prompt edit (v5) flipped UNRELATED fields (claim_status, issue_type) on case_003 — impossible from that edit, so the model had been sampling nondeterministically (temperature never set), meaning our ±5-10 deltas were partly noise. temperature=0 gives reproducible runs (satisfies "Deterministic where possible"), a reproducible output.csv, and clean A/B. NOTE: the cache key does NOT encode temperature, so the existing `.cache` (sampled at default temp) must be cleared and key configs re-baselined at temp=0 before their numbers are trusted.
- **Severity rubric edit (v5) documented INEFFECTIVE; reverted → v4** / Alt: keep v5 as ship / Reason: EXP-008 (clean temp=0) initially concluded v5 was inert because crack cases stay severity=high (model labels crack as glass_shatter, so the rule never fires). However, the clean v4-t0 baseline (EXP-010) showed v4 out-scores v5 — claim_status 85 vs 80, valid_image 95 vs 85 — proving the v5 severity refinement was NOT inert: it regressed headline accuracy. Reverted to v4 as ship config. v5 is skipped; next confirmed-effective prompt change → v6. See also CLUE.md §7 (versioning discipline).
- **Prompt versioning discipline: version bumps ONLY on confirmed-effective changes (CLAUDE.md §7)** / Alt: bump on every attempt (prior practice) / Reason: v5 was bumped at experiment-time before scoring confirmed it — it regressed claim_status 85→80, turning the version number into a tombstone rather than a live config. Rule: (1) experimental prompt changes happen on a branch WITHOUT bumping PROMPT_VERSION; (2) only after experiment scores confirm improvement does the version bump happen (on merge to main). This keeps version numbers monotonic and meaningful — every version in main's history represents an adopted improvement. The v5 experiment artifact (report + scoreboard row) remains in history as evidence of what was tried and why it was rejected.
- **REJECT an OCR stage and REJECT an injection-rule layer (no text-signal pipeline)** / Alt: add tesseract OCR as an independent authenticity/injection signal; OR a deterministic "text_instruction_present ⇒ block auto-support / force manual_review" rule / Reason: EXP-011 validated this at near-zero cost before building anything. (1) The core VLM already reads in-image text BETTER than tesseract — it emits `text_instruction_present` / `non_original_image` where warranted, while tesseract missed the handwritten "approve this claim" entirely; OCR would add a weaker, redundant signal. (2) A case_020 ablation (mask the injection note, re-run) left the verdict UNCHANGED (still over-supports), proving the over-support is PERCEPTUAL — the model misperceives the box as torn-open — NOT injection-driven; the prompt's security framing already contains the injection (verdict is injection-invariant). An injection rule would only fix 020 by luck (right verdict, wrong reason) and risk legit supported cases. The contradicted ceiling (014/020) is the SAME perceptual ceiling as issue_type — not a text-signal gap. Text/OCR is ruled OUT; do not revisit.
- **ADOPT a deterministic rule layer for the DERIVABLE risk flags** / Alt: keep trusting the model's risk_flags emission; OR try to fix risk_flags via more prompt edits / Reason: EXP-012 — per-flag analysis showed risk_flags(73) loss is over-emission of flags that are FUNCTIONS of structured inputs, not perception: `manual_review_required` had 7 false positives, and `user_history_risk` is 100% determined by the joined `history_flags` (20/20). Deriving them deterministically in `pipeline._derive_risk_flags()` (after clamp) lifted risk_flags micro-F1 73→80 with ZERO extra API calls and NO regression on any other field, and satisfies the contract's "deterministic where possible". This is the rule layer planning.md §6 always specified but never built. Prompt-editing was rejected as the fix because the errors aren't a prompt/perception gap — they're a calibration gap the model can't self-correct (it over-emits regardless). This is NOT a prompt change, so PROMPT_VERSION stays v4. Perceptual residual flags (`non_original_image`, `claim_mismatch`, `damage_not_visible`) left to the model — not derivable.

## Current planned architecture (reflects the decisions above)

MVP = candidate A (single VLM call does extraction + visual judgment). Target =
candidate B (two-stage decoupled). A's prompt becomes B's stage [2], so the
evolution wastes little. The eval framework is already built and is decoupled
(it consumes the output CSV; it does not run the VLM).

```
INPUTS (dataset/)                       SHARED
  claims.csv ─────────┐                 code/schema.py
  user_history.csv ──┐│                   (14-col order + allowed values;
  evidence_req.csv ─┐││                    imported by BOTH system & eval)
  images/ ────────┐ │││
                  ▼ ▼▼▼
            ┌──────────────┐   ╔════════ PER-CLAIM PIPELINE (static DAG) ════════╗
            │ Loader/join  │──►║ [1] claim extraction                            ║
            │ +img resolver│   ║      MVP(A): folded into the single VLM call    ║
            └──────────────┘   ║      B   : cheap text-only LLM → target         ║
                               ║            {object,part,issue,severity,multi[]} ║
            ┌──────────────┐   ║                      │                          ║
            │ Cache        │◄─►║ [2] VLM visual assessment (all row images)      ║
            │ key = imgset │   ║      issue? part visible? quality?              ║
            │   + promptver│   ║      authenticity? in-image text?               ║
            └──────────────┘   ║          │                                      ║
                               ║   (opt) deterministic CV helpers ─┐ (on demand) ║

            └──────────────┘   ║      (risk context only, never overrides pixels)║
                               ║          │                                      ║
                               ║ [4] fusion + schema clamp (rule layer)          ║

                               ╚══════════════════════╪══════════════════════════╝
                                                      ▼
                                                  output.csv
                                                      │
                                                      ▼
                            ┌──────────────────────────────────────────┐
                            │ EVAL FRAMEWORK  (code/evaluation/, built) │
                            │ validate schema + score vs sample truth   │

                            └──────────────────────────────────────────┘
```
